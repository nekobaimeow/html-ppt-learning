📐 核心公式
$$EAT = (1-p)T_{mem} + p T_{disk}$$
虚拟内存 (Virtual Memory)

操作系统为每个进程提供的独立的、连续的地址空间抽象。进程看到的虚拟地址不等于物理内存地址——由 MMU (Memory Management Unit) 硬件在每次内存访问时自动翻译。这个翻译过程对运行中的程序完全透明。

有效访问时间 (EAT)

$$EAT = (1-p) \times T_{\text{mem}} + p \times T_{\text{disk}}$$
p = 缺页率  |  Tmem ≈ 100ns  |  Tdisk ≈ 10ms

关键洞察：虚拟地址空间 ≠ 物理内存大小。一个 32 位进程"看到"4GB 地址空间，但可能只占用 200MB 物理内存——其余在磁盘上，按需换入。

没有虚拟内存的世界

❌ 进程直接操作物理地址——进程 A 的 bug 可能覆盖进程 B 的数据

❌ 必须全部加载到内存才能运行——8GB 程序在 4GB 机器上无法启动

❌ 物理内存碎片化——即使总空闲够用，找不到连续大块分配给新进程

✅ 进程隔离——每个进程看到独立地址空间，无法互相破坏

✅ 按需换页——只需加载当前需要的页面，其余留在磁盘

✅ 内存超售——多个进程可共享物理内存，系统总"分配量"可超物理上限

现实案例：游戏引擎

Unreal Engine 5 的 Nanite 虚拟几何系统需要处理数十亿个三角形——远超过 GPU 显存。它借鉴虚拟内存思想：只将当前视锥体内可见的几何微块加载到显存，远处或背面的数据留在系统内存/磁盘中。这是虚拟内存思想在图形领域的直接应用。

📊 没有虚拟几何 → 每帧 30M 三角形（卡顿）vs 有虚拟几何 → 每帧 300K 三角形（流畅 60fps）

📐 偏移计算
$$offset = vaddr \bmod 4096$$

CPU 发出虚拟地址

进程执行 mov eax, [0x7fff1234] — 这个地址是虚拟的，不是物理内存的某个位置

MMU 查询 TLB（快表）

TLB 是硬件缓存，存放最近翻译过的虚拟→物理地址映射。命中率 > 99%，仅需 ~1 CPU cycle

TLB miss → 遍历多级页表

MMU 从 CR3 寄存器取页表基址，逐级遍历 4 级页表（x86-64）。每级取 9 位虚拟地址做索引

缺页中断 (Page Fault)

PTE 的 Present 位 = 0 → CPU 触发缺页异常 → OS 处理程序从磁盘交换区读入页面 → 更新页表 → 重试指令

内存满了，然后呢？

❌ 进程请求新页面 → 物理内存已全部占满 → 必须换出一个旧页面

❌ 换错了后果严重：刚换出的页面马上又被访问 → 再次缺页 → 磁盘 I/O 开销 ×2

✅ 好的置换算法：换出"以后最不可能用到"的页面，最大化缓存命中率

场景：数据库服务器的内存压力

PostgreSQL 的 shared_buffers 设为 8GB（物理内存 16GB）。高峰期 50 个并发查询扫描不同表，总工作集 30GB——远超缓存大小。Buffer Manager 必须持续换出数据页。换出策略直接影响查询延迟：

📊 好的置换 → 命中率 95% → avg query 5ms  |  差的置换 → 命中率 60% → avg query 80ms

直觉陷阱："最近最少用"一定是最优吗？

大部分时候是的——这是 LRU (Least Recently Used) 的基础。但如果在循环扫描一个大数组（如 SELECT COUNT(*) FROM big_table），刚换出的页面马上又要被访问——LRU 在这种情况下反而表现最差。扫描污染 (scan pollution) 是 LRU 的软肋。

📐 缺页率公式
$$p = \frac{\text{faults}}{\text{accesses}}$$

每个物理页配一个访问位 (Reference Bit)

MMU 硬件在页面被访问时自动将该位置 1。OS 可以读/清零此位。这是 Clock 算法唯一需要的硬件支持。

时钟指针循环扫描物理页

需要换出页面时，指针从当前位置开始扫描：若 Ref=1 → 清零并跳过（给第二次机会）；若 Ref=0 → 选中换出。

两级 Clock 处理脏页

干净页可直接丢弃（磁盘有副本）。脏页必须先写回磁盘。改进的 Clock 给脏页额外的"存活机会"——降低写回开销。

Linux 实际使用：两链表 LRU

Active list（热页）+ Inactive list（冷页）。页面在两条链表间迁移。需要换出时从 Inactive 尾部取——这就是 Linux 的页面回收 (Page Reclaim) 机制。

/* Clock 页面置换算法伪代码 */ page_t* clock_select_victim(void) { while (1) { page_t *p = &frames[clock_hand]; if (p->referenced == 0) return p; // 找到牺牲页 p->referenced = 0; // 清零，给第二次机会 clock_hand = (clock_hand + 1) % NUM_FRAMES; } } /* 优势: O(1) 扫描，不需要时间戳链表 相比真正 LRU: 省掉每次访问更新链表的开销 */

📐 地址转换
$$paddr = frame \times 4096 + offset$$
分页 (Paging)

将物理内存和虚拟地址空间都划分为固定大小的块——页（Page，通常 4KB）。虚拟地址由页号和页内偏移组成。MMU 通过页表将虚拟页号映射到物理页框号。分页对程序完全透明。

分页地址翻译

$$\text{Physical Address} = \text{Frame} \times 2^{12} + \text{Offset}$$
虚拟地址 32-bit → 高 20-bit = 页号，低 12-bit = 偏移（4KB 页）
页号通过页表查得物理页框号 → 与偏移拼接得到物理地址

📐 段地址转换
$$paddr = \text{base}[S] + offset$$
分段 (Segmentation)

将地址空间按逻辑含义划分为可变大小的段——代码段、数据段、栈段、堆段。每个段有独立的基址 (base) 和界限 (limit)。分段对程序员可见——早期的段寄存器 (CS, DS, SS, ES) 就是分段机制的体现。

分段地址翻译

$$\text{Physical Address} = \text{Segment Table}[S].\text{base} + \text{Offset}$$
段号 S 由段寄存器提供 → 查段表得基址 + 长度 → 检查 offset < limit → 基址 + offset = 物理地址
越界访问 → 段错误 (Segmentation Fault) ——这个错误名就从这里来的！

分页 vs 分段：六维度对比

维度分页 (Paging)分段 (Segmentation)
划分依据固定大小（4KB）逻辑含义（代码/数据/栈）
外部碎片无有（需紧凑化）
内部碎片每页平均浪费 2KB无（精确分配）
共享粒度页级（CoW, mmap）段级
保护机制页级 (R/W/X bits)段级（天然权限边界）
程序透明性完全透明程序员需管理段
磁盘 I/O 友好固定页大小 = 批量 I/O段大小不固定，I/O 复杂
现代 OS 采用Linux/Windows/macOS 主方案仅保留兼容模式

💡 为什么现代 OS 选择分页？

分页的无外部碎片特性在长时间运行的服务器上至关重要——分段运行数天后碎片化严重，需要 Stop-The-World 紧凑化。分页的固定大小也使磁盘换入换出高度优化：一个 I/O 操作正好一页。同时 Copy-on-Write、mmap 共享、NUMA 迁移等机制都建立在页粒度的基础上。

🏛️ 实际上 Intel x86 使用了段页式——段机制用于兼容性（所有段基址 = 0），页机制做真正的内存管理。
